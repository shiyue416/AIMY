# 🧠 举一反三机制 — RAG + 推理引擎
> 不更新模型权重，获得模型权重级别的泛化能力

---

## 核心问题

```
Payload库缺点：只匹配已知模式
  见过 ?url=http://127.0.0.1
  不知道 ?endpoint=//127.0.0.1 也是SSRF

模型权重优点：见过100个SSRF案例后
  能推断出第101种变体
```

**但不需要改权重就能获得这个能力。**

---

## 解法：RAG + 少样本推理

```
传统Payload库：
  Payload → 匹配 → 找到/找不到

举一反三方式：
  Payload库（历史案例）→ RAG检索相关案例
      ↓
  Claude：「这里有个参数接受URL，我见过3个类似案例：
           案例1用//绕过，案例2用0x7f，案例3用%40@
           → 生成10个变体payload测试」
      ↓
  发现新洞 → 写回案例库（飞轮）
```

**权重没变，但实现了举一反三。**

knowledge_graph.py   # 存历史案例

### 串联方式

```
新目标发现可疑参数
    ↓
knowledge_graph 检索：「历史上类似参数怎么被打的」
    ↓
reasoning_engine 推断：「这个上下文可能有哪些变体」
    ↓
payload_mutator 生成：「基于历史生成新变体」
    ↓
测试 → 发现新洞 → 写回 knowledge_graph
```

```json
{
  "vuln_type": "SSRF",
  "param_name": "url",
  "context": "image proxy endpoint",
  "working_payload": "http://169.254.169.254/latest/meta-data/",
  "bypass_technique": "cloud metadata",
  "variations_tried": ["//127.0.0.1", "0x7f000001", "%40@"],
  "target_tech": "nginx + python",
  "impact": "AWS credentials leaked"
}
```

有了结构化案例，Claude 看到新目标时：
> "这个参数上下文像图片代理，历史案例里有6个类似的，
> 其中2个用云元数据bypass成功 → 优先试这两个payload"

---

## 与纯Payload库的差距

```
纯Payload库：
  100个SSRF payload → 命中率取决于库的覆盖度

RAG + 推理：
  100个历史案例 → 能生成 1000个上下文相关变体
               → 覆盖率随案例积累指数增长
```
