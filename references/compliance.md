# 合规与合法红线

在提交任何 finding 前,本文件必须被读取并逐条比对。任一项不通过 → finding 撤回 / 重测。

---

## ❌ 绝对禁止

- **出 scope 资产**:目标不在 Phase 1 已确认的 in-scope 列表 → 立即停手 + 自报告给项目方。
- **取走他人 PII**:仅证明"可访问",不实际下载 / 转储他人数据。已不慎获取的立即销毁并记录。
- **持续负载 / DoS / 大流量**:仅发 1–3 个 PoC 包证明可触发,严禁压测 / 自动化连发。
- **修改他人数据**:即使发现可写也只在**自己控制的对象**上验证。不准对他人订单 / 账户 / 内容 / 资料做写操作。
- **生产钓鱼 / 社工**:不做。
- **未复现猜测**:不得提交"理论上可能存在"的 finding。必须有 HTTP 包 / 截图 / 视频证据。
- **拖第三方 SaaS**:出 scope 的云资产、CDN、托管服务一律不测。
- **横向移动到内网**:即使从 SSRF / RCE 触达内网,仅证明"可达",不进一步横向。

---

## ✅ 必做

- 测试 header 自标:`X-Bug-Bounty: <你的 handle>` 或项目要求的标记。
- 越权 / IDOR 用**自己控制的两个账号**自演,不要拿真实用户 ID 试探。
- SSRF / XXE / RCE 外带用**自己的 OOB 域名**(interactsh / 自建 DNSLog),不要用公共 / 他人 DNSLog。
- 提交前用 `references/templates/report-submission.md` 自查三段式结构是否完整。
- 提交前对照 CVSS 4.0 vector 给评级,不要凭感觉写 High/Critical。
- 提交后保留所有原始包 + 截图 ≥ disclosure window 结束。

---

## 撤回触发条件

下列任一情况立即向项目方撤回 finding:
- 事后发现资产其实 out-of-scope
- 复现步骤中包含取走的他人 PII
- PoC 流量超过 1–3 个包还在继续
- 影响范围被夸大(评级 vs 实际可证明影响不符)

---

## 与 Phase 4 的边界

Phase 4 的 playbook 中每个攻击章节都有自己的"不要做的事"小节。本文件是**通用红线**,playbook 内的限制是**类型特定红线**,二者都要遵守。冲突时取更严格的一项。
